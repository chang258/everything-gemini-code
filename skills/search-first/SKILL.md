# Search First Workflow

此技能系統化了「在實作之前搜尋現有解決方案」的工作流程，避免重複造輪子。

## 觸發時機
- 開始一個可能已有現有解決方案的新功能時。
- 需要添加新的依賴項或整合外部服務時。
- 準備建立新的工具類 (Utils)、輔助函數 (Helpers) 或抽象層之前。

## 標準工作流程

1. **需求分析**: 定義所需功能，識別語言/框架限制。
2. **平行搜尋**: 使用 `google_web_search` 或 `codebase_investigator` 搜尋相關工具。
3. **評估**: 根據功能、維護狀況、許可證 (MIT/Apache) 及依賴規模進行評分。
4. **決策**: 
   - **完全匹配**: 直接採用/安裝。
   - **部分匹配**: 擴展現有方案或編寫薄包裝器 (Wrapper)。
   - **無合適方案**: 自訂建構。

## 決策矩陣
- **高品質現成工具**: **採用** — 優先於自寫程式碼。
- **基礎良好但功能不足**: **擴展** — 繼承現有工具並添加自訂邏輯。
- **未找到合適工具**: **建構** — 確保自訂程式碼具備高度複用性。

## 執行指令與工具

### 快速檢查清單
在開始寫程式前，請反思：
1. 這是一個常見問題嗎？ (搜尋 npm/PyPI/GitHub)
2. 目前的 Gemini Skills 中是否有相關工具？ (檢查 `/skills list`)
3. 是否有現成的 GitHub 模板或範例可參考？

### 深度研究代理
對於複雜功能，請調用 sub-agent 進行研究：
- 使用 `codebase_investigator` 尋找專案內是否已有類似實作。
- 使用 `google_web_search` 尋找外部最佳實踐。

## 反模式
- **直接編碼**: 未經搜尋即開始編寫通用的工具函數。
- **依賴膨脹**: 為了微小功能引入巨大的函式庫。
- **忽略現有 Skill**: 不檢查系統已具備的自動化能力。
